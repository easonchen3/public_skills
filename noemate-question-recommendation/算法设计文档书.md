# NOEMate 公共 Skill 能力设计文档

## ——问题推荐（Question Recommendation）

---

## 1. 背景与目标

NOEMate 是华为 iMaster NAIE 27.0 的核心组件，面向 NOC 运维工程师、规划/维护/优化/运营经理等角色，提供智能问答、故障诊断、网络监控、投诉分析、容量规划等能力。当前系统已经具备以下基础能力：

* AICOService：智能体核心服务，负责对接 LUI 后台服务
* 短期上下文：
  1）用户原始问题
  2）用户改写问题
  3）知识片段
  4）识别用户意图（与 Skill 对应）
  5）任务规划与 Skill 执行结果
* 记忆系统：支持高频相似问题存储与 TopN 返回，并支持 embedding 能力
* Skill Engine：负责 Skill 执行

现需新增一个​**公共 Skill 能力：问题推荐**​，在智能体主回答之后输出推荐问题，帮助用户快速进入下一步查询、分析、诊断或规划。

### 1.1 设计目标

问题推荐能力需要满足以下目标：

1. 在主回答后输出 **Top3 推荐问题**
2. 推荐结果直接返回**纯问题列表**
3. 推荐问题必须**结合当前上下文**
4. 推荐问题必须**基于当前意图 / Skill**
5. 推荐问题应优先服务于“下一步任务推进”，而不是简单做相似问句扩写
6. 方案需满足**首版本快速上线**
7. 方案需兼容 27.1 Skill 扩展，并具备持续演进能力

### 1.2 版本约束

当前版本存在明确约束：

* 首版本只输出 **Top3**
* 只返回​**纯问题列表**​，不输出解释、分类、来源
* **只允许调用一次大模型**
* 不要求新增复杂前置结构化链路
* 当前上下文中**无法稳定直接获得关键实体、结果标签**
* Skill 场景持续扩展，**不能依赖模板穷举**
* 记忆系统可返回高频相似问题 Top3\~5，并支持 embedding 能力

## 2. 设计原则

### 2.1 推荐的是“下一步问题”，不是“相似问题”

NOEMate 用户的目标是完成任务，包括知识理解、数据查询、故障定位、投诉分析、网络监控、容量规划等。因此推荐问题的核心目标不是“相似问法推荐”，而是：

> 基于当前问题和当前回答，判断用户下一步最可能、最有价值继续追问的问题。

### 2.2 方案必须适应现网长尾场景

现网问题问法开放、场景复杂、持续扩展，纯模板方案难以覆盖；同时当前阶段无法稳定依赖结构化实体和结果标签，因此首版本应优先选择：

> 对输入依赖最弱、对长尾场景泛化能力最强的方案。

### 2.3 首版本要在性能、精度、扩展性之间取平衡

由于只能调用一次大模型，不能采用多阶段 LLM 流水线。因此首版本方案必须：

* 前处理尽量轻
* 不依赖额外复杂中间能力
* 通过 Skill 约束和记忆先验增强来提升可控性
* 通过后处理保证输出质量

### 2.4 当前方案是首阶段能力，不是终态

首版本核心目标是快速跑通、验证价值、沉淀日志和反馈数据。后续再逐步引入轻量结构化、排序优化和行为数据学习。

* CN\_NetworkPlanning

## 4. 问题定义与方案选择

### 4.1 问题定义

给定当前完整会话上下文 $C$、当前 Skill $S$、记忆系统返回的历史高频问题先验 $M$，生成 Top3 推荐问题 $Q_1,Q_2,Q_3$，要求：

1. 与当前问题和主回答强相关
2. 符合当前 Skill 语义边界
3. 能推动用户进入下一步任务
4. 尽量符合真实用户追问方式
5. 输出为纯问题列表

形式化表示为：

$$
Q^* = \arg\max_{Q \in \mathcal{Q}} P(Q \mid C, S, M)
$$

其中：

* $C$：当前完整上下文
* $S$：当前意图 / Skill
* $M$：记忆系统提供的历史高频问题先验

### 4.2 方案比选

#### 方案A：模板 / 规则主导

优点：

* 可控性强
* 解释性好
* 实现直接

缺点：

* 场景不可穷举
* 长尾问题覆盖差
* Skill 扩展时维护成本高
* 依赖结构化实体和结果标签，当前条件不满足

#### 方案B：候选召回 + 排序

优点：

* 是常见工业化范式
* 可逐步演进到数据驱动

缺点：

* 需要稳定候选空间
* 需要较多高质量标注和行为数据
* 当前首版本场景扩展快，候选空间不稳定
* 一次模型调用限制下，不适合设计为复杂多阶段链路

#### 方案C：Skill 约束下的上下文生成 + 记忆增强

优点：

* 不依赖模板穷举
* 不依赖实体标签显式抽取
* 对场景扩展适应性最强
* 满足一次模型调用约束
* 与现有上下文兼容度高

缺点：

* 显式可解释性弱于规则方案
* 需要通过 Skill 约束、提示设计和后处理增强可控性

### 4.3 方案结论

首版本选择 ​**方案C：基于当前完整上下文 + 当前 Skill 约束 + 记忆高频问题增强 + 单次大模型直接生成 Top3**​。

这不是“简单把问题丢给模型”，而是在当前版本约束下，对性能、精度、扩展性做出的最优平衡。

## 5. 总体方案设计

### 5.1 总体思路

问题推荐能力设计为一个公共能力模块，在主回答完成后统一调用。推荐主链路如下：

1. 读取当前完整上下文
2. 获取记忆系统高频相似问题 Top3\~5
3. 根据当前 Skill 注入推荐偏好约束
4. 单次调用大模型生成 Top3 推荐问题
5. 使用轻量后处理做去重和重复过滤
6. 返回纯问题列表

### 5.2 总体架构

#### 上游依赖

* AICOService：提供问题、改写问题、知识片段、意图、规划、执行结果、最终回答
* 记忆系统：返回高频相似问题 Top3\~5，并提供 embedding 能力
* Skill Engine：提供 Skill 执行结果

#### 推荐能力模块

推荐能力包含三部分：

1. **上下文组装器**
   * 读取并压缩当前会话上下文
   * 组装给大模型的输入上下文
2. **Skill 约束配置器**
   * 根据当前 Skill 注入不同的推荐偏好约束
   * 不是模板库，而是 Skill 的推荐偏好画像
3. **推荐生成与后处理器**
   * 单次调用大模型生成 Top3
   * 去重、过滤与当前问题高度重复的问题
   * 输出纯问题列表

## 6. 输入上下文设计

### 6.1 使用已有上下文，不引入重型前置处理

首版本直接使用现有系统已具备的上下文信息，不新增复杂抽取链路。输入上下文包括：

1. 用户原始问题
2. 用户改写问题
3. 当前意图 / Skill
4. 检索知识片段
5. 任务规划摘要
6. Skill 执行结果摘要
7. 智能体最终回答
8. 记忆系统高频相似问题 Top3\~5

### 6.2 为什么不强依赖关键实体和结果标签

虽然关键实体（如 KPI、时间窗、MSISDN、Peer、网元）和结果标签（如是否已有根因、是否已有影响分析）长期来看有价值，但当前上下文无法直接稳定提供。若将方案建立在这些字段之上，会增加额外前置链路、复杂度和时延，不利于首版本落地。

因此首版本采用：

* 由模型对完整上下文做**隐式理解**
* 不将实体与结果标签作为前提
* 后续根据真实日志再逐步显式结构化增强

### 6.3 上下文压缩原则

由于只有一次大模型调用，需要控制输入长度。上下文组装时建议：

* 原始问题保留
* 改写问题保留
* 当前 Skill 必保留
* 知识片段只保留与当前回答最相关的摘要
* 任务规划只保留高层摘要，不保留全部中间 Action
* 执行结果保留关键信息
* 最终回答保留完整或压缩版摘要
* 记忆问题最多保留 Top3\~5

## 7. Skill 约束设计

### 7.1 设计原则

Skill 约束不是模板，不要求穷举问法，而是给模型注入“当前 Skill 下什么是高价值下一步问题”的偏好。其作用是：

* 限制推荐语义空间
* 避免模型发散
* 保证推荐与当前 Skill 匹配
* 为 27.1 Skill 扩展提供统一扩展位

### 7.2 27.0 / 27.1 Skill 推荐偏好

#### 7.2.1 CN\_KnowledgeQA

推荐应优先围绕：

* 知识点在异常场景中的表现
* 对应华为产品实现、网元、版本、配置查看
* 运维落地方法
* 关联告警、KPI、案例

不应优先推荐：

* 与知识点无关的泛监控问题
* 直接高风险执行动作

#### 7.2.2 CN\_DataAgent / CN\_NetworkMonitoring

推荐应优先围绕：

* 当前查询结果进一步下钻
* 异常来源、对象、区域、接口、Peer 等维度展开
* 同期对比、趋势分析
* 业务影响和关联告警/资源变化

不应优先推荐：

* 纯知识解释问题
* 与查询结果无关的泛排障动作

#### 7.2.3 CN\_FaultSpiri

推荐应优先围绕：

* 故障影响范围
* 根因进一步定位
* 证据补充（告警、日志、性能、对端）
* 恢复后的验证或值守监控

不应优先推荐：

* 脱离当前故障上下文的泛问题
* 纯知识介绍型问题

#### 7.2.4 CN\_CompSpiri

推荐应优先围绕：

* 同时段是否扩散到更多用户
* 所属网络环节定位
* 信令、日志、业务证据
* 是否需要进一步跟踪或追加诊断

不应优先推荐：

* 与投诉个体无关的宏观统计问题
* 容量规划类问题

#### 7.2.5 CN\_NetworkPlanning（27.1）

推荐应优先围绕：

* 长周期趋势
* 资源增长和区域差异
* 容量瓶颈与风险
* 扩容和规划建议

不应优先推荐：

* 与规划无关的即时故障处理问题

## 8. 记忆系统接入设计

### 8.1 记忆系统的定位

记忆系统不作为“最终推荐输出器”，而作为：

> 历史高频追问先验增强器

它的作用不是直接展示记忆问题，而是帮助模型理解：

* 类似场景下用户常见追问方向
* 真实用户语言风格
* 高频下一步问题模式

### 8.2 记忆系统输入与输出

输入建议基于：

* 用户改写问题
* 当前 Skill
* embedding 向量

输出：

* 相似场景高频问题 Top3\~5

### 8.3 记忆系统的使用方式

将记忆系统返回的问题作为模型输入中的“参考问题列表”。在 Prompt 中明确约束：

* 这些问题是类似场景下的高频问题，仅供参考
* 不要求逐字复用
* 应优先生成与当前上下文最匹配、最适合作为下一步问题的问题

### 8.4 embedding 的使用

embedding 主要用于后处理阶段：

* 过滤与当前问题高度近似的问题
* 过滤 Top3 内部高度重复的问题

## 9. 推荐生成设计

### 9.1 核心生成逻辑

推荐问题生成采用单次大模型调用，输入：

* 当前原始问题
* 当前改写问题
* 当前 Skill
* 知识片段摘要
* 任务规划摘要
* Skill 执行结果摘要
* 智能体最终回答
* 记忆系统高频问题 Top3\~5
* 当前 Skill 的推荐偏好约束

输出：

* 3 个最合适的后续推荐问题

### 9.2 模型需要完成的四项任务

#### 1）理解当前任务阶段

例如当前是：

* 知识解释阶段
* 数据查询阶段
* 异常发现阶段
* 根因定位阶段
* 结果验证阶段
* 规划分析阶段

#### 2）理解当前回答已完成到哪一步

例如：

* 已解释了原理，但未落地到运维
* 已发现异常，但未分析影响范围
* 已做初步判断，但未给证据
* 已给出统计结果，但未展开趋势分析

#### 3）结合当前 Skill，判断什么是合理下一步

例如：

* KnowledgeQA 更应引导到异常场景、产品实现、配置查看
* Fault 更应引导到影响范围、根因、证据和验证
* Monitoring 更应引导到下钻、对比和影响分析

#### 4）参考高频追问，生成自然用户风格的问题

即在符合上下文和 Skill 的前提下，尽量贴近真实用户语言风格

### 9.3 输出约束

模型输出必须满足：

* 固定输出 3 条
* 每条都是完整问题句
* 不输出解释、原因、分类
* 不重复当前问题
* 不输出明显泛化、无意义的问题

## 10. 后处理设计

### 10.1 去重

对 Top3 推荐问题做字符串去重和 embedding 相似度去重，避免重复表达。

### 10.2 当前问题近似过滤

若推荐问题与当前原问题 / 改写问题高度相似，则进行过滤或要求重生成。首版本如不做重生成，可从记忆问题中选择最合适候选补齐。

### 10.3 数量补齐

若过滤后不足 3 条，则从记忆系统返回问题中选取与当前问题不重复、且符合当前 Skill 的问题补齐。

### 10.4 语言一致性

推荐问题应与当前会话语言一致，中文优先中文，英文问题优先英文，避免中英混杂。

---

## 11. 端到端流程设计

### 11.1 主流程

#### Step 1：主回答完成

AICOService 完成改写、意图识别、知识检索、任务规划、Skill 执行与最终回答生成。

#### Step 2：收集推荐输入

读取：

* 原始问题
* 改写问题
* 当前 Skill
* 知识片段摘要
* 任务规划摘要
* 执行结果摘要
* 最终回答

#### Step 3：调用记忆系统

使用改写问题和当前 Skill 获取相似场景高频问题 Top3\~5。

#### Step 4：组装模型输入

按推荐任务要求组织完整上下文，并动态注入当前 Skill 的推荐偏好。

#### Step 5：单次调用大模型

生成 Top3 推荐问题。

#### Step 6：后处理

做去重、近似过滤、数量补齐。

#### Step 7：返回结果

输出纯问题列表，展示于主回答下方。

## 12. 方案合理性与技术取舍

### 12.1 为什么不是模板主导

模板适合场景收敛、问法有限、输入结构化稳定的任务。当前 NOEMate 的实际特点是：

* Skill 持续扩展
* 用户问题开放
* 现网长尾场景多
* 上下文丰富但不结构化

因此模板最多只能作为补充，不适合做主方案。

### 12.2 为什么不是复杂召回 + 排序主导

传统召回排序式架构需要：

* 稳定候选空间
* 足够训练数据
* 可复用的结构化特征
* 较复杂的链路设计

当前阶段不满足这些前提，同时受“一次大模型调用”限制，不适合首版本采用。

### 12.3 为什么选择 Skill 约束下的生成式方案

因为该方案在当前条件下具备最优综合性：

* 对模板和结构化依赖最低
* 泛化能力最强
* 可直接利用已有上下文
* 满足一次模型调用约束
* 容易兼容 27.1 Skill 扩展
* 工程落地速度最快

### 12.4 为什么这不是“简单 Prompt 方案”

该方案本质上是：

> 基于上下文、Skill 约束和历史高频问题先验的受约束生成系统

其逻辑闭环完整，包括：

* 明确的问题建模
* 备选方案比选
* 受约束下的最优取舍
* 可分阶段演进的架构路径

## 13. 测试与评测设计

### 13.1 评测目标

问题推荐不是单一分类问题，因此不能只看单一准确率。首版本评测重点是：

1. 是否贴当前上下文
2. 是否符合当前 Skill
3. 是否是合理下一步问题
4. 是否自然、非重复
5. 性能是否满足要求

### 13.2 评测集设计

#### 13.2.1 离线评测集

基于当前已收集的真实用户问题场景构造样本，按 Skill 分桶：

* KnowledgeQA
* DataAgent / NetworkMonitoring
* FaultSpiri
* CompSpiri
* NetworkPlanning（27.1）

每条样本包含：

* 用户原问题
* 改写问题
* 当前 Skill
* 当前回答摘要
* 记忆系统返回问题
* 强推荐问题集合
* 可接受问题集合
* 不应推荐问题集合

这里不要求唯一金标，而采用集合标注方式，更适合生成式任务。

#### 13.2.2 真实场景覆盖范围

优先覆盖你提供的以下高价值场景：

* 3GPP Attach 流程、无线建立连接结构与功能
* 今天有哪些故障或隐患导致业务受损
* 主机负荷、存储负荷查询
* Attach success rate dropped at XX time
* 某号码在某时段 VoLTE 掉话根因分析
* 脚本生成与校验
* 过去24小时接口信令流量异常突增
* 多种用户数统一统计
* 当前号码挂载在哪个 MME / SGW / PGW
* 创建跟踪、采集 debug 日志、命令行对齐
* 一年期 VoNR KPI 趋势分析
* 哪个 Peer 导致 traffic spike
* error code abnormal trend
* license 利用率与容量预测
* 多 EMS 数据合并与趋势汇报

### 13.3 评测维度

#### 13.3.1 Top3 可接受率

Top3 中属于“强推荐问题集合”或“可接受问题集合”的比例。

#### 13.3.2 Skill 一致率

推荐问题是否符合当前 Skill 的典型语义边界。

#### 13.3.3 下一步价值得分

由领域专家判断推荐问题是否真正推动任务继续完成。

#### 13.3.4 相关性得分

推荐问题与当前问题和当前回答的相关程度。

#### 13.3.5 重复率

包括：

* 与当前问题的重复率
* Top3 内部重复率

#### 13.3.6 性能指标

* 推荐总耗时
* 大模型调用耗时
* 后处理耗时

### 13.4 人工评测维度

建议按 1\~5 分打分，维度包括：

* 相关性
* 下一步价值
* Skill 一致性
* 自然性
* 多样性

首版本重点看前三项。

### 13.5 线上评测指标

上线后重点监控：

* 推荐点击率
* 推荐采纳率（点击后真实发送）
* 推荐后有效追问转化率
* 推荐后用户自行改写率
* 推荐链路时延
* 不良推荐反馈率

## 16. 风险与应对

### 16.1 风险1：生成问题与当前问题重复

应对：

* 引入字符串与 embedding 双重相似度过滤
* 对当前原问题和改写问题同时做近似过滤

### 16.2 风险2：推荐问题发散，不贴当前 Skill

应对：

* 强制注入 Skill 偏好约束
* 在评测中单独跟踪 Skill 一致率
* 针对问题 Skill 补充约束文案

### 16.3 风险3：推荐问题过泛、无执行价值

应对：

* Prompt 中明确约束“优先下一步高价值问题”
* 后处理过滤空泛表达
* 在人工评测中重点标注“下一步价值”

### 16.4 风险4：记忆问题误导模型

应对：

* 在 Prompt 中把记忆问题定义为“参考”
* 不允许模型机械照搬
* 不把记忆问题直接作为最终输出

### 16.5 风险5：新 Skill 扩展后推荐质量不稳

应对：

* Skill 偏好配置与主框架解耦
* 新 Skill 上线时同步扩评测样本
* 先扩偏好配置，再基于日志优化

## 17. 结论

问题推荐能力的首版本，建议设计为：

> **基于当前完整上下文 + 当前 Skill 约束 + 记忆高频问题增强 + 单次大模型直接生成 Top3 推荐问题**

该方案的核心价值在于：

1. ​**满足首版本性能约束**​：只调用一次大模型
2. ​**满足业务精度要求**​：推荐是“下一步问题”，不是简单相似问句
3. ​**不依赖模板穷举**​：适应现网长尾场景和 Skill 扩展
4. ​**不依赖前置结构化字段**​：适合当前上下文实际情况
5. ​**工程可快速落地**​：可直接依托现有 AICOService、记忆系统、Skill Engine
6. ​**后续演进清晰**​：27.1 可平滑扩到 NetworkPlanning 和 OMC 动态数据查询，后续可逐步引入轻结构化与数据驱动优化

因此，该方案是当前阶段在**性能、精度、扩展性和交付速度**之间的最优平衡方案。
